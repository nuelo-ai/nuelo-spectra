# Phase 52.1: Delete and Rename Collection - Context

**Gathered:** 2026-03-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Add delete and rename actions for collections in the workspace UI. Users can rename a collection (name + description) and delete a collection (with cascade warning) from both the collection list page and the collection detail page. No new collection fields or schema changes beyond what's needed for these two actions.

</domain>

<decisions>
## Implementation Decisions

### Entry point — where actions are accessible
- **Both the collection card (list page) and the collection detail page**
- On the card: always-visible kebab menu (⋮) button in the **top-right corner** of the card (not hover-only)
- On the detail page: same kebab menu (⋮) in the **page header**, next to the collection name/title
- Both menus contain the same two items: "Rename" and "Delete"

### Rename UX
- **Reuse `CreateCollectionDialog` in edit mode** — pre-fill with current `name` and `description`, change dialog title to "Edit Collection", submit button says "Save"
- Do not create a separate dialog component — adapt the existing one with an optional `collection` prop for edit mode
- After successful save: **invalidate TanStack Query** collections list + detail cache (same pattern as existing mutations)
- No optimistic update — wait for server confirmation

### Delete confirmation
- Confirmation dialog shows: **collection name bolded** + cascade warning: "This will permanently remove all signals, reports, and detection runs in this collection."
- Submit button: **red destructive `variant="destructive"`**, labeled **"Delete"**
- During deletion: button is **disabled and shows "Deleting..."** (uses `isPending` pattern, same as CreateCollectionDialog)
- Cancel button: labeled "Cancel"

### Post-delete navigation
- Deleted from **detail page**: `router.push('/workspace')` after successful deletion
- Deleted from **list page** (card kebab): **invalidate collections query** — card disappears on next render
- Both paths: show **success toast** — `toast.success("Collection deleted")` using existing sonner pattern

</decisions>

<specifics>
## Specific Ideas

- The "Collection limit reached" dialog in `CreateCollectionDialog` references deleting a collection — after this phase ships, users will actually be able to do that
- The delete cascade is already in place at the DB level (signals, pulse_runs, reports all have `ondelete="CASCADE"` on `collection_id` FK) — no DB migration needed

</specifics>

## Existing Code Insights

### Reusable Assets
- `frontend/src/components/workspace/create-collection-dialog.tsx`: Reuse with edit mode — add optional `collection?: CollectionListItem` prop; when present, pre-fill fields, change title/button, call PATCH instead of POST
- `frontend/src/hooks/useWorkspace.ts`: Add `useUpdateCollection` (calls `PATCH /collections/{id}`) and `useDeleteCollection` (calls `DELETE /collections/{id}`) hooks — follow existing `useCreateCollection` mutation pattern
- `frontend/src/components/workspace/collection-card.tsx`: Add kebab menu using shadcn `DropdownMenu` in top-right corner of `CardHeader`; wrap card Link to prevent navigation when clicking the menu
- `frontend/src/app/(workspace)/workspace/collections/[id]/page.tsx`: Add kebab menu to the detail page header row (next to collection name)
- `backend/app/services/collection.py`: `CollectionService.delete_collection()` exists at line ~255 — the service method is ready, just needs the router endpoint
- `backend/app/routers/collections.py`: `PATCH /{collection_id}` (update) already exists — only need to add `DELETE /{collection_id}` endpoint
- `toast` from `sonner` already imported in detail page — use for success/error feedback

### Established Patterns
- TanStack Query invalidation: `queryClient.invalidateQueries({ queryKey: ['collections'] })` after mutation — same as `useCreateCollection`
- Mutation loading state: `mutation.isPending` to disable button and show "..." label
- Error handling: catch API errors and call `toast.error(message)` — same pattern as existing workspace hooks
- Backend router: `DELETE /{collection_id}` follows same ownership-check pattern as `PATCH /{collection_id}` (verify `user_id` ownership, return 404 if not found, 204 on success)

### Integration Points
- `backend/app/routers/collections.py`: add `DELETE /{collection_id}` endpoint (204 No Content), calling `CollectionService.delete_collection()`
- `frontend/src/components/workspace/collection-card.tsx`: add DropdownMenu with Rename/Delete items; pass callbacks from parent
- `frontend/src/components/workspace/collection-list.tsx`: thread `onRename` and `onDelete` callbacks down to `CollectionCard`
- `frontend/src/app/(workspace)/workspace/page.tsx`: manage `renameTarget` and `deleteTarget` state, render edit/delete dialogs
- `frontend/src/app/(workspace)/workspace/collections/[id]/page.tsx`: add kebab menu to header, manage rename/delete dialog state, `router.push('/workspace')` on delete success
