Rewrite the Continue to Chat button's onClick handler (currently lines 232-267 of FileUploadZone.tsx) to be fault-tolerant.

The current handler has TWO critical bugs:
1. The optional context POST (lines 234-243) has `return` in its catch block, which BLOCKS all subsequent actions (dialog close, tab open, sidebar refetch) when the POST fails.
2. Steps 2-6 (invalidateQueries, refetchQueries, openTab, onUploadComplete, reset state) have NO try/catch. Any unhandled error causes silent abort via unhandled promise rejection.

Replace the entire onClick async function body with this structure:

```typescript
onClick={async () => {
  try {
    // Step 1: Save user context (non-blocking, fire-and-forget)
    // Context is OPTIONAL -- failure must never block dialog close or tab open
    if (userContextInput.trim() && uploadedFileId) {
      try {
        await apiClient.post(`/files/${uploadedFileId}/context`, {
          context: userContextInput,
        });
      } catch (error) {
        console.error("Failed to save user context:", error);
        toast.error("Context could not be saved, but your file is ready.");
        // Do NOT return -- proceed with dialog close and tab open
      }
    }

    // Step 2: Refresh sidebar file list
    try {
      queryClient.invalidateQueries({ queryKey: ["files"], exact: true });
      await queryClient.refetchQueries({ queryKey: ["files"], exact: true });
    } catch (error) {
      console.error("Failed to refresh file list:", error);
      // Sidebar will catch up via refetchInterval fallback
    }

    // Step 3: Open file tab
    if (uploadedFileId) {
      openTab(uploadedFileId, uploadedFileName);
    }
  } catch (error) {
    console.error("Continue to Chat handler error:", error);
    toast.error("Something went wrong. Your file was uploaded successfully.");
  } finally {
    // ALWAYS close dialog and reset state, regardless of any errors above
    if (onUploadComplete) {
      onUploadComplete();
    }
    setUploadStage("idle");
    setProgress(0);
    setUploadedFileId(null);
    setUploadedFileName("");
    setAnalysisText(null);
    setUserContextInput("");
    hasTransitioned.current = false;
  }
}}
```

KEY DESIGN DECISIONS:
- `finally` block guarantees dialog close + state reset. This is non-negotiable -- the user must never be stuck in a dialog that won't close.
- Context POST failure shows a toast warning but does NOT return. The context is documented as optional (FILE-05).
- `exact: true` on invalidateQueries/refetchQueries prevents unnecessarily refetching the summary query (only refetch the file list).
- Each step has its own try/catch so one failure doesn't cascade to the next.
- Console.error on every catch for debugging via browser DevTools.

**File 1: useFileManager.ts -- Add refetchInterval to useFiles**

In the `useFiles` function (lines 13-25), add `refetchInterval: 10000` to the query options. This provides an independent fallback: even if the button handler fails to trigger a refetch, the sidebar will discover new files within 10 seconds. Change:

```typescript
export function useFiles() {
  return useQuery({
    queryKey: ["files"],
    queryFn: async (): Promise<FileListItem[]> => {
      const response = await apiClient.get("/files/");
      if (!response.ok) {
        throw new Error("Failed to fetch files");
      }
      return response.json();
    },
    refetchOnWindowFocus: true,
  });
}
```

To:

```typescript
export function useFiles() {
  return useQuery({
    queryKey: ["files"],
    queryFn: async (): Promise<FileListItem[]> => {
      const response = await apiClient.get("/files/");
      if (!response.ok) {
        throw new Error("Failed to fetch files");
      }
      return response.json();
    },
    refetchOnWindowFocus: true,
    // Independent fallback: sidebar discovers new files within 10s
    // even if button handler fails to trigger manual refetch
    refetchInterval: 10000,
  });
}
```

WHY 10 seconds: Frequent enough that users see files appear reasonably quickly after upload, but not so aggressive that it causes unnecessary network traffic. The primary path (button handler explicit refetch) provides instant updates; this is purely a safety net.

Currently (line 35) only destructures `data` and `isLoading`:

```typescript
const { data: files, isLoading } = useFiles();
```

Change to also destructure `isError`:

```typescript
const { data: files, isLoading, isError } = useFiles();
```

Then add an error state branch in the rendering logic. Currently (lines 87-166) the conditional is:

```
isLoading ? skeleton : files && files.length > 0 ? file list : empty state
```

Add an error branch AFTER the isLoading check and BEFORE the files check. The full conditional should become:

```tsx
{isLoading ? (
  <div className="space-y-2 p-2">
    {[1, 2, 3].map((i) => (
      <div key={i} className="skeleton h-12 rounded-lg" />

  <div className="flex flex-col items-center justify-center py-12 px-4 text-center opacity-60">

  // ... existing file list rendering (keep as-is)

  // ... existing empty state (keep as-is)
)}
```

WHY error state: Without this, a failing GET /files/ shows "No files yet" -- misleading the user into thinking they have no files. With the error state, users see "Failed to load files" and "Retrying automatically..." (which is true because of refetchInterval). This is critical for debugging and user trust.
