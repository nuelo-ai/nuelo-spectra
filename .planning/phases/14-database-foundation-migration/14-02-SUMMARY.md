## Decisions Made

1. **Use session_files for checkpoint migration**: Initially planned to use chat_messages table to get file->session mappings, but discovered files with no messages but with checkpoints. Changed to use session_files table which covers ALL files.

2. **Preserve orphaned checkpoints**: Found 21 checkpoints pointing to deleted files. These are harmless (never accessed) and will age out naturally. No need to delete them.

3. **Timezone-aware datetime handling**: Fixed data migration to use DateTime(timezone=True) to match PostgreSQL TIMESTAMP WITH TIME ZONE column type. Without this, asyncpg raised "can't subtract offset-naive and offset-aware datetimes" error.

4. **Checkpoint downgrade is irreversible**: Checkpoint migration downgrade() raises NotImplementedError with guidance to restore from backup. Reversing thread_id changes is not practical without backup.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed timezone-aware datetime handling in data migration**
- **Found during:** Task 2, running data migration
- **Issue:** Data migration failed with "can't subtract offset-naive and offset-aware datetimes" error when inserting into chat_sessions. The files.created_at column returns timezone-aware datetime objects, but the table() construct used DateTime() without timezone=True, causing PostgreSQL type mismatch.
- **Fix:** Changed DateTime to DateTime(timezone=True) in files_table and chat_sessions table definitions
- **Files modified:** backend/alembic/versions/ad17e35bacd0_migrate_file_conversations_to_sessions.py
- **Commit:** 1646978

**2. [Rule 1 - Bug] Fixed checkpoint migration to handle files with no messages**
- **Found during:** Task 2, verification of checkpoint migration
- **Issue:** Initial checkpoint migration logic used chat_messages table to get file->session mappings. This missed files that have checkpoints but no messages yet (conversations started but no messages sent). Discovered 21 orphaned checkpoints but verified they're for deleted files, not active files.
- **Fix:** Changed checkpoint migration to use session_files table instead of chat_messages table, ensuring ALL files (with or without messages) get their checkpoints migrated
- **Files modified:** backend/alembic/versions/2792e8318130_migrate_langgraph_checkpoints_and_.py
- **Commit:** 1646978

## Issues Encountered

**1. Alembic autogenerate detected LangGraph table drops**
- **Issue:** Running `alembic revision --autogenerate` detected checkpoints, checkpoint_blobs, checkpoint_writes, checkpoint_migrations tables as "removed" and generated drop table commands
- **Root cause:** LangGraph tables are managed externally by langgraph-checkpoint-postgres, not by our Alembic migrations. Autogenerate compares database state to ORM models and sees these tables as "extra"
- **Fix:** Manually edited migration to remove all checkpoint table drop commands from upgrade() and checkpoint table create commands from downgrade()
- **Resolution:** Schema migration runs cleanly without touching LangGraph tables

**2. Orphaned checkpoints for deleted files**
- **Issue:** After migration, found 21 checkpoints with file-based thread_ids. Investigation revealed these point to files that no longer exist in the files table.
- **Root cause:** Files were deleted at some point, but LangGraph checkpoints are not automatically cleaned up (by design for recovery purposes)
- **Decision:** Preserved these orphaned checkpoints. They are harmless (never accessed since file doesn't exist) and will age out naturally. Deleting them would be unnecessarily aggressive.
- **Verification:** Confirmed that all 18 ACTIVE files have their checkpoints migrated to session-based format (0 active file-based checkpoints remain)
