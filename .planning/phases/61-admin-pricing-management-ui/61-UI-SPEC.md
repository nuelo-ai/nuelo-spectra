## Page Layout Contract

Page: `/billing-settings` (admin-frontend)

Layout wrapper: `max-w-2xl mx-auto space-y-6` (existing pattern)

### Section Order (D-02)

1. **Page Header** — "Billing Settings" title + description subtitle
2. **Monetization Card** — Toggle + description + Stripe readiness checklist when disabled (D-12)
3. **Subscription Pricing Card** — List of subscription tiers in view mode. Each tier row: tier name, current price display, config default hint below. Edit button per tier opens modal (D-06). "Reset to Defaults" button at card bottom (D-11)
4. **Credit Packages Card** — Grid/list of credit package cards in view mode. Each card: name, price, credits, display_order, active badge, Stripe Price ID (read-only). Edit button per package opens modal (D-06). "Reset to Defaults" button at card bottom (D-11)

### View Mode Layout — Subscription Tier Row

```
[Tier Name]                              [Edit button]
$XX.XX / month
Default: $YY.YY                          (muted hint text)
```

- Tier name: `text-sm font-medium`
- Price display: `text-lg font-semibold`
- Default hint: `text-xs text-muted-foreground` — format "Default: $XX.XX" (D-04)
- Edit button: `variant="outline" size="sm"` with Lucide `Pencil` icon

### View Mode Layout — Credit Package Card

```
┌─────────────────────────────────────┐
│ [Package Name]        [Active badge]│
│ $XX.XX · XXX credits                │
│ Display order: N                    │
│ Stripe: price_xxxxx     [Edit btn]  │
└─────────────────────────────────────┘
```

- Package name: `text-sm font-medium`
- Price and credits: `text-sm` — format "$XX.XX · XXX credits"
- Display order: `text-xs text-muted-foreground`
- Stripe Price ID: `text-xs font-mono text-muted-foreground` — truncate if long
- Active badge: `<Badge variant="default">Active</Badge>` or `<Badge variant="secondary">Inactive</Badge>`
- Edit button: `variant="outline" size="sm"` with Lucide `Pencil` icon
- Config default hint below price: `text-xs text-muted-foreground` — "Default: $XX.XX · XXX credits" (D-04)

## Interaction Contracts

### Click-to-Edit Flow (D-06)

1. User sees pricing data in **view mode** (read-only display)
2. User clicks **Edit** button on a specific tier or package
3. **Edit modal** opens with current values pre-filled
4. User modifies fields
5. User clicks **Save Changes** in modal
6. **PasswordConfirmDialog** opens with impact description
7. User enters password and clicks **Confirm**
8. Request sent to backend with data + password
9. On success: modal closes, toast success, data refetched
10. On 401 (wrong password): inline error below password field "Incorrect password", dialog stays open
11. On other error: toast error, dialog stays open

### Reset Flow (D-11)

1. User clicks **Reset to Defaults** at bottom of Subscription Pricing or Credit Packages card
2. **PasswordConfirmDialog** opens with destructive variant:
   - Impact text explains what will happen
   - Confirm button uses `variant="destructive"`
3. User enters password and clicks **Confirm Reset**
4. Request sent to backend with scope + password
5. On success: toast success, data refetched to show new values
6. On 401: inline error, dialog stays open
7. On other error: toast error, dialog stays open

### Loading States

- Initial page load: `Skeleton` components in each card (existing pattern)
- Save/Reset in progress: button shows "Saving..." / "Resetting..." with `disabled` state
- PasswordConfirmDialog confirm button: "Confirming..." with `disabled` state during request

### Error States

- API fetch failure: centered error message with destructive text (existing pattern from billing-settings page)
- Individual field validation: handled in edit modals before triggering password dialog

## Public Frontend Changes (UI-01, UI-02)

### Plan Selection Page (`/settings/plan`)

Per D-15: No frontend changes needed. The page already renders dynamically from the `/subscriptions/plans` API. Backend changes (D-14) to make the endpoint dynamic are sufficient.

### Billing Page (`/settings/billing`)

Per D-16: The billing page already renders credit packages from `useCreditPackages()` hook dynamically. The `CreditPackageCard` component renders from database data. No frontend template changes needed -- the existing grid `grid-cols-1 sm:grid-cols-3 gap-4` handles dynamic counts. If any hardcoded credit package references exist, they must be removed in favor of database-driven rendering.
