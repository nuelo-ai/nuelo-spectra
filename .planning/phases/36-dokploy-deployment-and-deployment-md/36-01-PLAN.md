<task type="checkpoint:human-action" gate="blocking">
  <name>Task 1: Create Dokploy PostgreSQL database and generate SECRET_KEY</name>
  <action>
    Perform these steps in order:

    **Step 1 — Create PostgreSQL database in Dokploy:**
    1. Open Dokploy UI → Databases → New → PostgreSQL
    2. Name: `spectra-db`
    3. Set a strong password (save it — you will need it for DATABASE_URL)
    4. Click Create
    5. Once created, navigate to the database → Connection section
    6. Copy the "Internal Connection URL" — it will look like:
       `postgresql://spectra:PASSWORD@spectra-db-HASH:5432/spectra`
    7. Save this URL. You will need it in Task 3.

    **Step 2 — Generate SECRET_KEY (must be same for both backend services):**
    Run this command locally or on the server:
    ```bash
    python3 -c "import secrets; print(secrets.token_hex(32))"
    ```
    Save the output. This value goes into BOTH backend services as `SECRET_KEY`.

    **Step 3 — Construct DATABASE_URL for FastAPI (asyncpg prefix):**
    Take the Dokploy Internal Connection URL and change `postgresql://` to `postgresql+asyncpg://`:
    ```
    postgresql+asyncpg://spectra:PASSWORD@spectra-db-HASH:5432/spectra
    ```
    This is the value you will paste into the Environment tab for both backend services.
    The Dokploy-provided URL uses `postgresql://` — FastAPI's SQLAlchemy async engine requires `postgresql+asyncpg://`.
  </action>
  <verify>
    Confirm:
    - [ ] Dokploy shows spectra-db with status "Running"
    - [ ] You have saved the Internal Connection URL
    - [ ] You have generated a SECRET_KEY (64-character hex string)
    - [ ] You have the asyncpg-prefixed DATABASE_URL ready
  </verify>
  <done>PostgreSQL database is running in Dokploy, asyncpg DATABASE_URL is saved, and SECRET_KEY is generated and stored for use in both backend services.</done>
  <resume-signal>Type "db-ready" once the database is running and you have the DATABASE_URL and SECRET_KEY values saved.</resume-signal>

<task type="checkpoint:human-action" gate="blocking">
  <name>Task 2: Install Tailscale on Dokploy host and configure VPS cloud firewall</name>
  <action>
    Perform these steps in order. Both can be done in parallel (Tailscale on server, firewall in cloud console).

    **PART A — Install Tailscale on the Dokploy host server:**

    SSH into the Dokploy host server, then run:

    ```bash
    # Step 1: Install Tailscale
    curl -fsSL https://tailscale.com/install.sh | sh

    # Step 2: Find the Docker internal network subnet
    docker network inspect dokploy-network | grep -i subnet
    # Example output: "Subnet": "10.254.0.0/24"
    # Note the subnet — you will use it in the tailscale up command

    # Step 3: Bring up Tailscale with subnet routing
    # Replace <TAILSCALE_AUTH_KEY> with a key from https://login.tailscale.com/admin/settings/keys
    # Replace 10.254.0.0/24 with the actual dokploy-network subnet from Step 2
    sudo tailscale up \
      --ssh \
      --advertise-routes=10.254.0.0/24 \
      --auth-key=<TAILSCALE_AUTH_KEY> \
      --hostname=spectra-prod

    # Step 4: Verify Tailscale is running
    tailscale status
    # Should show: spectra-prod with your Tailscale IP (100.x.x.x)

    # Step 5: Get the server's Tailscale IP (you will need this for admin service env vars)
    tailscale ip -4
    # Example: 100.64.0.1
    # Save this IP — it becomes <TAILSCALE_IP> in admin service env vars
    ```

    **Step 6 — In Tailscale admin console (https://login.tailscale.com/admin/machines):**
    - Find "spectra-prod" machine
    - Click the "..." menu → "Edit route settings"
    - Enable the advertised subnet route (10.254.0.0/24)
    - This allows Tailscale clients to reach Docker containers via subnet routing

    **PART B — Configure VPS cloud firewall:**

    In your VPS provider's cloud console (DigitalOcean, Hetzner, AWS, etc.) — NOT on the server with UFW — add firewall rules to block admin ports. UFW alone does NOT work (Docker bypasses UFW).

    Rules to add (DENY inbound from 0.0.0.0/0):
    - Port 8001 TCP — admin backend (DROP / no response)
    - Port 3001 TCP — admin frontend (DROP / no response)

    The Tailscale IP range (100.64.0.0/10) should be allowed through. If your VPS cloud firewall doesn't support Tailscale IP ranges, simply blocking these ports from all public IPs is acceptable — Tailscale access goes through WireGuard (UDP port 41641) which bypasses the Docker host TCP stack.

    **Save the Tailscale IP** (from `tailscale ip -4`). This will be used as `<TAILSCALE_IP>` in admin service environment variables.
  </action>
  <verify>
    On the Dokploy host server:
    ```bash

    # Expected: shows spectra-prod with Tailscale IP

    # Subnet routes are advertised

    # From a Tailscale client device (after running: tailscale up --accept-routes):

    ```

<task type="checkpoint:human-action" gate="blocking">
  <name>Task 3: Deploy public backend to Dokploy with volume mount configured before first deploy</name>
  <action>
    CRITICAL ORDERING: Configure the volume mount BEFORE clicking Deploy for the first time. Uploads created before the volume is mounted live in the container's writable layer and will NOT be migrated to the volume.

    **Step 1 — Create Dokploy Application for public backend:**
    In Dokploy UI → Applications → New Application:

    General tab:
    - Name: `spectra-public-backend`
    - Source: Git (connect to your GitHub/GitLab account)
    - Repository: `your-org/spectra-dev`
    - Branch: `main` (or your production branch)
    - Build Type: `Dockerfile`
    - Dockerfile Path: `Dockerfile.backend`
    - Docker Context Path: `.` (repo root — required for COPY backend/ instructions)

    **Step 2 — Set environment variables (Environment tab):**
    Paste ALL of the following, substituting your actual values:
    ```
    DATABASE_URL=postgresql+asyncpg://spectra:PASSWORD@spectra-db-HASH:5432/spectra
    SPECTRA_MODE=public
    SECRET_KEY=<your-generated-64-char-hex>
    ALGORITHM=HS256
    ACCESS_TOKEN_EXPIRE_MINUTES=30
    REFRESH_TOKEN_EXPIRE_DAYS=30
    FRONTEND_URL=https://app.yourdomain.com
    CORS_ORIGINS=["https://app.yourdomain.com"]
    ADMIN_CORS_ORIGIN=
    APP_VERSION=v0.6.0
    ANTHROPIC_API_KEY=sk-ant-...
    OPENAI_API_KEY=
    GOOGLE_API_KEY=
    OLLAMA_BASE_URL=
    OPENROUTER_API_KEY=
    LANGSMITH_API_KEY=
    LANGSMITH_TRACING=
    LANGSMITH_PROJECT=
    E2B_API_KEY=
    TAVILY_API_KEY=
    SEARCH_DEPTH=basic
    SEARCH_DAILY_QUOTA=
    SMTP_HOST=
    SMTP_PORT=587
    SMTP_USER=
    SMTP_PASS=
    SMTP_FROM_EMAIL=
    SMTP_FROM_NAME=
    ENABLE_SCHEDULER=true
    ADMIN_EMAIL=
    ADMIN_PASSWORD=
    ```
    Notes:
    - DATABASE_URL must use `postgresql+asyncpg://` prefix (not `postgresql://`)
    - FRONTEND_URL and CORS_ORIGINS: use your actual public frontend domain (even if not yet deployed — you can update these later)
    - ADMIN_CORS_ORIGIN: leave empty for public mode
    - ENABLE_SCHEDULER=true: enables the credit reset scheduled job

    **Step 3 — Configure volume mount BEFORE deploying (Advanced tab → Mounts):**
    Click Advanced → Mounts → Add Mount:
    - Type: Volume
    - Volume Name: `spectra-uploads`
    - Mount Path: `/app/uploads`
    Click Save.

    **Step 4 — Deploy:**
    Click Deploy. Monitor the deployment logs.
    The container will:
    1. Wait for PostgreSQL to accept connections (pg_isready loop, max 60s)
    2. Run `alembic upgrade head` (creates all tables)
    3. Start uvicorn on port 8000

    The deployment log should show: "INFO: Application startup complete." without any Alembic errors.
  </action>
  <verify>
    After deployment completes:

    ```bash

    # Check migration ran (look for alembic in startup logs):
    docker logs <container-id> 2>&1 | grep -i alembic
    # Expected: "Running upgrade..." or "INFO  [alembic.runtime.migration] Running upgrade"

    # Check uvicorn started:
    docker logs <container-id> 2>&1 | grep -i "application startup complete"
    # Expected: "INFO: Application startup complete."

    # Verify volume is mounted at /app/uploads:
    docker inspect <container-id> | python3 -m json.tool | grep -A 5 "Mounts"
    # Expected: shows spectra-uploads volume mounted at /app/uploads

    # Health check via curl (from inside Dokploy server via internal Docker network):
    docker exec <container-id> curl -s http://localhost:8000/health
    # Expected: {"status": "ok"}
    ```

    Confirm:
    - [ ] Dokploy shows spectra-public-backend as "Running"
    - [ ] Deployment logs show Alembic migrations completed
    - [ ] `docker inspect` shows spectra-uploads volume mounted at /app/uploads
    - [ ] `/health` returns `{"status": "ok"}`
  </verify>
