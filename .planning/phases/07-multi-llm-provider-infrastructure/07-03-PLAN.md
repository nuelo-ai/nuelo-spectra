    **Add startup LLM validation to the FastAPI lifespan in `backend/app/main.py`.**

    Per user decision: Backend must refuse to start if any configured provider is unreachable or has invalid credentials. Validation must check both config correctness and connectivity.

    **Implementation approach:**

    Create the validation logic directly in `main.py` (keep it simple, avoid separate validator module to prevent circular imports per Research pitfall #3). Use lazy imports inside the lifespan function.

    Add a `validate_llm_configuration()` async function in `main.py` that performs:

    **Step 1: Config validation**
    - Load provider registry from `load_provider_registry()`
    - Load prompts config from `load_prompts()`
    - For each agent in prompts, get its `provider` (explicit or default)
    - Verify each agent's provider exists in the registry
    - If any agent references an unknown provider, raise with message: "Agent '{agent_name}' configured with provider '{provider}' which is not in llm_providers.yaml. Available providers: {list}"

    **Step 2: Collect active providers**
    - Build set of providers actually used by agents (not all providers in registry, only referenced ones)
    - This avoids requiring unused providers to be configured

    **Step 3: Connectivity validation per active provider**
    Use `httpx.AsyncClient` (already a project dependency) with 5-second timeout:

    - **anthropic**: Check `settings.anthropic_api_key` is non-empty. GET `https://api.anthropic.com/v1/models` with `x-api-key` header and `anthropic-version: 2023-06-01`. 401 = invalid key. Timeout = network issue.
    - **openai**: Check `settings.openai_api_key` is non-empty. GET `https://api.openai.com/v1/models` with Bearer auth. 401 = invalid key.
    - **google**: Check `settings.google_api_key` is non-empty. GET `https://generativelanguage.googleapis.com/v1/models?key={key}`. 400/403 = invalid key.
    - **ollama**: GET `{settings.ollama_base_url}/api/tags` (lists local models, no auth needed). ConnectError = Ollama not running. Timeout = network issue.
    - **openrouter**: Check `settings.openrouter_api_key` is non-empty. GET `https://openrouter.ai/api/v1/models` with Bearer auth. 401 = invalid key.

    **Error classification per user decision:**
    - **Temporary failures:** `httpx.TimeoutException`, `httpx.ConnectError` -> message includes "Check network connectivity" or "Check that {service} is running"
    - **Permanent failures:** 401/403 status, empty API key -> message includes "Check your {ENV_VAR} setting" or "Get an API key from {URL}"

    **Error message format (user-facing):**
    ```
    LLM Configuration Error: {provider} provider failed validation
    Agent(s) affected: {comma-separated agent names}
    Error: {specific, actionable message}
    ```

    **Logging (technical details):**
    ```python
    logger.error(f"LLM validation failed", extra={"provider": name, "error_type": ..., "status_code": ..., "agents": [...]})
    ```

    If `validate_llm_configuration()` raises, the exception propagates and FastAPI refuses to start. This is the desired behavior.

    **1. Add structured LLM call logging to `backend/app/agents/llm_factory.py`:**

    Per user decision: Log metadata for all LLM calls (timestamp, provider, model, agent, token count, latency, status) but NOT full prompts/responses. Use Python standard `logging` module with `json.dumps()` for structured output.

    Add an `invoke_with_logging()` async wrapper function:

    ```python
    import logging
    import json
    import time

    logger = logging.getLogger("spectra.llm")

    async def invoke_with_logging(
        llm: BaseChatModel,
        messages: list,
        agent_name: str,
        provider: str,
        model: str,
    ):
        """Invoke LLM with structured metadata logging.

        Logs: timestamp, provider, model, agent, latency, status.
        Does NOT log: full prompts or responses (LangSmith handles that).

        Args:
            llm: LangChain chat model instance
            messages: List of messages to send
            agent_name: Name of the calling agent (for logging context)
            provider: Provider name (for logging)
            model: Model name (for logging)

        Returns:
            LLM response

        Raises:
            LLMProviderError: Classified provider error with user-friendly message
        """
        start_time = time.time()

        try:
            response = await llm.ainvoke(messages)
            latency = time.time() - start_time

            # Log successful call metadata (NOT full content)
            logger.info(json.dumps({
                "event": "llm_call",
                "agent": agent_name,
                "provider": provider,
                "model": model,
                "status": "success",
                "latency_seconds": round(latency, 3),
            }))

            return response

        except Exception as e:
            latency = time.time() - start_time

            # Classify error type
            error_class = classify_llm_error(e, provider)

            # Log error metadata
            logger.error(json.dumps({
                "event": "llm_call",
                "agent": agent_name,
                "provider": provider,
                "model": model,
                "status": "error",
                "error_type": type(e).__name__,
                "error_class": error_class,
                "latency_seconds": round(latency, 3),
            }))

            raise
    ```

    Add a `classify_llm_error(error, provider)` function that returns a classification string:
    - `"network_error"` for connection/timeout errors (httpx.ConnectError, httpx.TimeoutException, ConnectionError, TimeoutError)
    - `"auth_error"` for authentication failures (check error message for "401", "403", "unauthorized", "invalid api key")
    - `"rate_limit"` for quota exceeded (check for "429", "rate limit", "quota")
    - `"model_not_found"` for invalid model (check for "404", "model not found", "not found")
    - `"provider_error"` for anything else

    This classification maps to user decision: "Distinguish failure types — Provide specific error messages."
