<decisions_made>

- Replaced next.config.ts rewrites with catch-all route handler (`app/api/[...slug]/route.ts`) ‚Äî rewrites have 3 fundamental problems for this app (SSE buffering, build-time BACKEND_URL, trailing slash stripping)
- Backend middleware to strip trailing slashes rather than trying to preserve them through proxy ‚Äî makes API work regardless of caller behavior
- `redirect_slashes=False` on FastAPI ‚Äî prevents 307 redirects that lose auth headers through any proxy
- Routes changed from `"/"` to `""` on prefixed routers (files, chat_sessions) ‚Äî matches paths after middleware strips trailing slashes
- Admin proxy preserves `/api/` prefix in path (`/api/admin/users` ‚Üí `BACKEND_URL/api/admin/users`), public proxy strips it (`/api/files` ‚Üí `BACKEND_URL/files`)
- For non-SSE responses, proxy reads full body via `arrayBuffer()` before creating Response ‚Äî avoids Content-Length mismatches from redirects
- Phase 34 plans are unaffected ‚Äî Dockerfiles don't use `next.config.ts` rewrites (the removed feature), they reference BACKEND_URL as build arg which the route handler reads at runtime
</decisions_made>

<context>
The Phase 33 "rewrite proxy" approach seemed correct in isolation but had 3 hidden problems that only surfaced during manual testing: trailing slash stripping (307 + auth loss), SSE buffering, and build-time BACKEND_URL. The catch-all route handler approach solves all 3. This also resolves the SSE concern flagged in STATE.md ‚Äî no need for NEXT_PUBLIC_BACKEND_URL fallback.

For Phase 34 Dockerfiles: the BACKEND_URL is now a runtime env var (read by the route handler at request time), NOT a build-time var. This is BETTER than the original rewrite approach for Docker ‚Äî you can change BACKEND_URL without rebuilding the image. The Phase 34 plans mentioned BACKEND_URL as a Docker build arg for rewrites, but that's no longer needed. The planner should be aware of this change.
</context>
