# Phase 51.1: Pipeline Refactor - Research

**Researched:** 2026-03-08
**Domain:** LangGraph multi-agent orchestration, Pydantic structured output, async detection UX
**Confidence:** HIGH

## Summary

This phase refactors the Pulse Agent from a monolithic 3-node LangGraph pipeline into a multi-agent orchestrator pattern where the Pulse Agent acts as a brain calling independent tool agents. The current `run_analyses_node` does everything inline (LLM candidate generation, code generation, validation, sandbox execution, result parsing) with manual JSON parsing and no structured output. The refactor breaks this into discrete agents (Coder, Validator, Interpreter, Visualization, Report Writer), each with Pydantic structured output via `with_structured_output()`.

The frontend changes are moderate: replace the full-page `DetectionLoading` overlay with an inline progress banner, add global toast notifications, keep Run Detection visible after completion with re-run confirmation, add "View Signal Report" button, change "Recent Signals" to "Signals Identified", and make signal cards clickable to navigate to Detection Results.

**Primary recommendation:** Keep the LangGraph StateGraph approach but restructure into orchestrator + tool agent functions. Reuse `validate_code()`, `extract_code_block()`, and `E2BSandboxRuntime` directly -- they are pure/stateless. Create new Pulse-specific agent functions (not reusing Chat's `coding_agent` which depends on `ChatAgentState` and `get_stream_writer()`).

### Pattern 1: Orchestrator with Tool Agent Functions

**What:** The Pulse Agent graph has more nodes, each representing a distinct agent responsibility. The orchestrator node (Pulse brain) drives the flow by producing structured instructions consumed by downstream tool nodes.

**When to use:** When a single pipeline needs multiple specialized LLM calls with different prompts and output schemas.

**Design:**
```python
# State gains new fields for each agent's output
class PulseAgentState(TypedDict):
    # ... existing fields ...
    signal_hypotheses: list[SignalHypothesis]      # From hypothesis node
    validation_results: list[ValidationResult]      # Per-signal coder+sandbox results
    business_findings: list[BusinessFinding]        # From interpretation node
    chart_instructions: list[ChartInstruction]      # From viz node
    report: ReportOutput                            # From report writer node

# Graph structure
# profile_data -> generate_hypotheses -> fan_out_validation -> interpret_findings -> generate_charts -> write_report
```

### Pattern 2: with_structured_output() for All LLM Calls

**What:** Use LangChain's `llm.with_structured_output(PydanticModel)` instead of manual JSON parsing.

**How it works:**
```python
from pydantic import BaseModel, Field

class SignalHypothesis(BaseModel):
    title: str = Field(description="Business-language signal title")
    severity_hint: Literal["critical", "warning", "info"]
    category: str
    code_instructions: str = Field(description="Prescriptive instructions for Coder")
    chart_hint: str

class HypothesisOutput(BaseModel):
    hypotheses: list[SignalHypothesis]

# Usage
structured_llm = llm.with_structured_output(HypothesisOutput)
result = await structured_llm.ainvoke(messages)  # Returns HypothesisOutput directly
```

**Eliminates:** All manual JSON parsing, markdown stripping, `content.startswith("```")` hacks.

### Pattern 3: Fan-out Validation with Per-Signal Loop

**What:** The current `asyncio.gather()` fan-out stays but each step becomes clearer: Coder generates code -> Validator checks it -> Sandbox executes -> Interpreter makes business finding -> Viz generates chart. All sequential per signal but signals processed in parallel.

### Pattern 4: Re-run Overwrite Flow

**What:** On re-run, delete existing signals for the collection's latest completed run before persisting new ones. Keep PulseRun records for audit.

```python
# In _run_pipeline, before persisting new signals:
await db.execute(
    delete(Signal).where(Signal.collection_id == collection_id)
)
# Then persist new signals as before
```

### Anti-Patterns to Avoid
- **Reusing Chat's `coding_agent` function:** It depends on `ChatAgentState`, `get_stream_writer()`, and Chat-specific prompt formatting. Create a new `pulse_coder_agent` function instead.
- **Passing raw data to interpretation LLM:** Coder's code must return pre-aggregated compact JSON. Never pass raw dataframes or large arrays.
- **Single mega-prompt:** Each agent step gets its own focused prompt and Pydantic output model.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON parsing from LLM | Manual regex/strip/parse | `with_structured_output()` | Handles retries, validation, edge cases |
| Code safety check | Custom validator | Existing `validate_code()` | Already handles AST analysis, allowlists |
| Code extraction | Custom regex | Existing `extract_code_block()` | Handles all markdown code block formats |
| Toast notifications | Custom event system | sonner or shadcn toast | Standard React toast pattern |

## Common Pitfalls

### Pitfall 1: with_structured_output() Provider Compatibility
**What goes wrong:** Not all providers support structured output the same way. Anthropic uses tool_use, OpenAI uses function_calling or json_mode.
**How to avoid:** LangChain abstracts this -- `with_structured_output()` handles provider differences. The pulse_agent currently uses Anthropic claude-sonnet-4-20250514 which fully supports it.
**Warning signs:** If switching to a provider that does not support tool_use, structured output may fail silently.

### Pitfall 2: Fan-out Timeout with Many Candidates
**What goes wrong:** 5-8 candidates each requiring LLM call + sandbox execution can take 3-5 minutes total.
**How to avoid:** Keep `PULSE_SANDBOX_TIMEOUT_SECONDS=300` per sandbox. Consider limiting to max 6 candidates. The background `asyncio.create_task()` pattern already handles this.

### Pitfall 3: Stale DB Session After Long Pipeline
**What goes wrong:** The current code already handles this (re-fetches `pulse_run` after pipeline). Keep this pattern in the refactor.
**How to avoid:** Always re-fetch ORM objects after long async operations within the same session.

### Pitfall 4: DetectionLoading Full-Page Takeover
**What goes wrong:** Currently `if (detectionStatus === "running") return <DetectionLoading />` replaces the entire page. This is what we are fixing.
**How to avoid:** Remove the early return, replace with an inline progress banner component within the Overview tab content.

### Pitfall 5: Signal Overwrite Without Cleanup
**What goes wrong:** Re-running detection creates duplicate signals if old ones are not deleted.
**How to avoid:** Delete existing signals for the collection before persisting new ones. Keep PulseRun records.

## Code Examples

### Pydantic Models for Structured Output

```python
# New models for each agent step
class SignalHypothesis(BaseModel):
    """Output from Pulse brain: a single signal hypothesis."""
    title: str = Field(description="Business-language title, no jargon")
    severity_hint: Literal["critical", "warning", "info"]
    category: str = Field(description="anomaly, trend, correlation, distribution")
    code_instructions: str = Field(description="Prescriptive: 'Calculate X, aggregate by Y, return Z'")
    chart_hint: str

class HypothesisOutput(BaseModel):
    """Structured output from hypothesis generation."""
    hypotheses: list[SignalHypothesis]

class BusinessFinding(BaseModel):
    """Interpretation of sandbox results in business language."""
    title: str
    finding: str = Field(description="Business-language finding for leadership")
    severity: Literal["critical", "warning", "info"]
    evidence: SignalEvidence

class ChartInstruction(BaseModel):
    """Instructions for visualization agent."""
    chart_type: Literal["bar", "line", "scatter"]
    title: str
    description: str = Field(description="What business story the chart tells")

class ReportOutput(BaseModel):
    """Structured report from Report Writer."""
    executive_summary: str
    signal_sections: list[dict]
    content: str = Field(description="Full markdown report")
```
