## Chart Spinner — 3 rounds of fixes

### Round 1 (commit b1fc1c6) — FAILED
- Added chartReady state + spinner overlay + onReady callback
- Test result: No spinner shown, lag still present
- Root cause diagnosed: React 18 batching collapses setChartReady(false) and setChartReady(true) into a single render when both fire in the same effect flush. Spinner never commits to DOM.

### Round 2 — PARTIALLY FIXED but two new issues
Fixes applied:
1. `signal-detail-panel.tsx` — added `key={signal.id}` to `<SignalChartRenderer>`
   - Forces full remount on signal switch → chartReady resets to false from useState(false) initial value, BEFORE any effects run → spinner shows in first render
2. `ChartRenderer.tsx` — wrapped Plotly.react() in `setTimeout(0)`
   - Intended to push Plotly past current paint so browser can commit spinner first

Test result:
- Spinner DID show (key fix worked)
- BUT spinner overlapped analysis text below the chart card
- Screen still hung while Plotly ran

### Round 3 — COMMITTED, NOT YET TESTED (current state)
Fix 3: `signal-chart-renderer.tsx` — outer wrapper div: `className="relative"` → `className="relative min-h-[300px]"`
  - Rationale: Without min-h, the relative container has zero height when chart is opacity-0 (no content to expand it). The absolute-positioned spinner uses inset-0 which fills zero height, causing it to overflow into the analysis text below.
  - Also removed redundant `h-[300px]` from the spinner inner div (inherits from parent via inset-0 now).

Fix 4: `ChartRenderer.tsx` — replaced `setTimeout(0)` with double `requestAnimationFrame`
  - Rationale: setTimeout(0) does not guarantee the browser has painted before it fires. Double-rAF is the standard technique: first rAF schedules before next paint, second rAF schedules before the paint after that — guarantees browser has committed spinner to screen.
  - Updated cleanup: `cancelAnimationFrame(raf1); cancelAnimationFrame(raf2)` (was `clearTimeout(timer)`)

Files modified (all committed):
- `frontend/src/components/workspace/signal-detail-panel.tsx`
- `frontend/src/components/workspace/signal-chart-renderer.tsx`
- `frontend/src/components/chart/ChartRenderer.tsx`

### Known remaining issue after Round 3 (A→B→A navigation)
With key={signal.id}, every signal switch fully remounts SignalChartRenderer. If you navigate A→B→A:
- chartReady resets to false (spinner shows) ✓
- plotlyJSON recomputes (same string content as before, since same signal.id and data)
- ChartRenderer's useEffect sees data unchanged (same string) → does NOT re-run → onReady never fires → spinner stays forever
This is a known edge case. Whether it manifests in practice depends on whether Plotly's bdata-encoded JSON produces the same string. TO INVESTIGATE after main spinner test passes.

### Root cause of silent failures
The E2B runtime captures BOTH `result.stderr` (list of stderr lines) AND `result.error.traceback` (full Python traceback). But the current viz failure logging only captures `result.error` (a truncated dict):

```python
# pulse.py lines 443-447 — CURRENT:
error_info = str(viz_result.error)[:300] if viz_result.error else "unknown"
logger.warning(
    "  [%s] Viz sandbox failed (attempt %d): error=%s",
    title, viz_attempt + 1, error_info,
)
# MISSING: viz_result.stderr and viz_result.error.traceback are never logged
```

Without stderr/traceback in the logs, we cannot see the actual Python error causing the crash.

### What was ruled out
- Module allowlist: scipy, plotly, plotly.graph_objects, plotly.express ALL in allowlist.yaml — code validation would pass
- Plotly availability in sandbox: 5 signals successfully generated charts, so plotly IS installed
- Column name issues in coder code: generated_code uses exact correct column names with spaces
- chart_data format (bdata): Frontend Plotly.js 3.x handles this — not a frontend issue for the 5 passing signals
- Token limits: Haiku has 200k input context and 4096 output tokens — viz code (50-100 lines) is well within output limit

### Hypotheses for viz failure (cannot confirm without logs)
H1 (likely): Viz sandbox execution crashes with a Python exception. Possible causes:
  - For scatter (correlation): viz LLM uses scipy.stats.linregress for regression line — if E2B viz sandbox doesn't have scipy pre-installed (unlike coder sandbox), ImportError
  - For bar (seasonal): month ordering logic uses pd.Categorical or .reindex(['Jan','Feb',...]) — if Month values in actual data don't match expected strings exactly, KeyError or empty result
  - For line (volatility): date parsing combining Year (int64) + Month (str "Jan") via pd.to_datetime — common failure mode, produces NaT or errors

H2 (possible): Viz code runs successfully but last line of stdout is not the chart JSON. If viz code has extra print statements or the JSON print is not the final line, viz_result.stdout[-1] is not valid JSON → json.loads fails → outer except catches it → chart_data stays None.

H3 (less likely): analysis_text passed to viz prompt is very long for complex analyses, causing the viz prompt to be large → Haiku produces subtly malformed code that passes validation but fails at runtime.
