# Phase 51.1: Pipeline Refactor - Context

**Gathered:** 2026-03-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Refactor the Pulse Agent pipeline to a proper multi-agent orchestration architecture where the Pulse Agent acts as the brain/orchestrator calling independent tool agents. Enforce Pydantic structured output on all LLM calls. Add UI improvements for async detection UX (inline progress, toast notifications, re-run flow). No changes to Chat Session agents or pipeline.

</domain>

<decisions>
## Implementation Decisions

### Pipeline architecture — Pulse Agent as orchestrator
- Pulse Agent is the **brain/orchestrator** that drives the entire detection flow
- It calls independent tool agents to achieve its goal, not monolithic nodes
- **Zero changes to Chat Session agents** — existing `coding_agent`, `code_checker_node`, `execute_in_sandbox`, `visualization_agent_node` are untouched
- Research determines whether to reuse underlying utility functions (`validate_code()`, `extract_code_block()`, `E2BSandboxRuntime`) with Pulse-specific prompts, or create fully new Pulse-specific agent workers
- All generated code saved to DB for audit trail (potential "View Code" feature on Signal detail page)

### Pipeline flow (per detection run)
1. **Profile Data** — generate deep profile if not cached on File model (skip if `file.deep_profile` exists)
2. **Signal Hypothesis** — Pulse Agent analyzes profiles, identifies N signal candidates
3. **Per-Signal Validation Loop** (for each candidate):
   a. Pulse Agent writes a **specific, prescriptive instruction** for what to validate (concrete metrics, aggregations, thresholds)
   b. **Coder Agent** (independent tool) — writes Python code per instruction
   c. **Code Validator** (independent tool) — AST + safety check
   d. If valid → **Sandbox execution** — runs the code, returns pre-aggregated results (not raw data)
   e. **Pulse Agent interprets findings** — dedicated LLM call with compact aggregated data → business-language finding
   f. **Visualization Agent** (independent tool) — generates relevant chart telling the business story
4. **Report Writer Agent** (independent tool) — after all signals collected, writes executive-friendly summary with facts, tables at its discretion

### Business language enforcement
- All findings, signal titles, and reports are written in **business terminology for leadership** — no statistical jargon
- Example: "High Revenue Concentration Risk — Top 5% of customers generate 82% of revenue" NOT "Z-score of 3.4 exceeds critical threshold"
- Pulse Agent's system prompt must enforce that its instructions to the Coder are **specific and prescriptive** about what metrics to compute and return
- Coder's code returns **pre-aggregated summaries** (compact JSON), never raw data rows — keeps interpretation call lightweight on tokens

### Structured LLM output
- **Pydantic structured output on ALL Pulse LLM calls** — use LangChain's `with_structured_output()` with Pydantic models
- No manual JSON parsing, no markdown stripping, no raw string manipulation
- Separate Pydantic models for each agent output: signal candidates, validation results, business findings, chart instructions, report structure

### Separate interpretation call
- After sandbox results return, Pulse Agent makes a **dedicated LLM call** to interpret findings in business language
- This is separate from validation — validation focuses on data correctness, interpretation focuses on business storytelling
- Data passed to interpretation is compact (pre-aggregated by Coder's code), not the full dataset
- Two Pydantic models: one for sandbox result structure, one for business finding output

### Async detection UX
- **Inline progress banner on Overview page** — user stays on the page, can still see and interact with existing content
- **Global toast notification** when detection completes — appears on any page in the app (Chat, Files, etc.), clicking navigates to Collection's Detection Results
- Run Detection button + optional `user_context` text input **always visible** (not hidden after detection completes)
- "Recent Signals" text changed to **"Signals Identified"**
- **"View Signal Report" button** above Files section — primary blue button with icon, prominent and noticeable
- Signal click on Overview page **navigates to Detection Results page** (`/workspace/collections/[id]/signals`) with that signal pre-selected

### Re-run & overwrite flow
- **Confirmation dialog with credit cost warning**: "Running detection again will replace your current signals and report. This will cost N credits. Continue?"
- **Keep all PulseRun history in DB** — audit trail preserved, previous runs shown in activity feed
- **UI shows only the latest run's signals/report** — no version switching in v0.8
- History display is a future feature (not in current requirements)
- 409 Conflict still enforced for concurrent runs on same collection
- User can provide new `user_context` for each re-run (text input stays visible)

### Claude's Discretion
- Whether to reuse Chat Session utility functions or create fully independent Pulse agent workers (based on research)
- Exact LangGraph graph structure and node wiring for the orchestrator pattern
- Pydantic model field designs for each agent's structured output
- System prompt content for Pulse Agent, Coder, Validator, Visualization, and Report Writer
- Inline progress banner visual design (consistent with Hex.tech palette)
- Toast notification implementation approach (global state vs event bus)
- How to structure the "View Code" data in the Signal model for future display

</decisions>

<specifics>
## Specific Ideas

- Pulse Agent should act like a senior business analyst: form hypotheses, commission specific analyses, interpret results for a non-technical audience
- Instructions from Pulse Agent to Coder must be prescriptive: "Calculate X, aggregate by Y, return Z" — not vague "analyze this data"
- Chart should tell the business story: "Top 5% customers generating 80% of sales" not just "bar chart of customer revenue"
- Report Writer has discretion on format — can include tables, bullet points, executive summary structure as it sees fit
- Generated code is saved for audit — may be displayed in Signal detail view in a future iteration

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `validate_code()` (`backend/app/agents/code_checker.py`): AST-based safety validation — pure function, no state dependency, directly reusable
- `extract_code_block()` (`backend/app/agents/coding.py`): extracts Python from markdown — pure function, directly reusable
- `E2BSandboxRuntime` (`backend/app/services/sandbox/e2b_runtime.py`): sandbox execution with `data_files` support — directly reusable
- `PulseAgentOutput` / `SignalOutput` (`backend/app/schemas/pulse.py`): existing Pydantic schemas — to be extended/refactored for structured output
- `profile_files_in_sandbox()` (`backend/app/agents/pulse_profiler.py`): deep profiling script — reusable as-is
- `PulseService` (`backend/app/services/pulse.py`): lifecycle orchestration — refactor to use new pipeline
- `build_pulse_graph()` (`backend/app/agents/pulse.py`): current 3-node graph — to be replaced with orchestrator pattern

### Established Patterns
- Per-agent YAML config: `prompts.yaml` entries with provider, model, temperature, max_tokens, system prompt
- `asyncio.create_task()` for background pipeline execution
- Service layer with static methods (`PulseService`, `CollectionService`)
- `get_stream_writer()` for SSE events (Chat pipeline) — may not apply to Pulse background tasks

### Integration Points
- `backend/app/agents/pulse.py`: complete rewrite — new orchestrator architecture
- `backend/app/services/pulse.py`: update `_run_pipeline()` to use new graph
- `backend/app/schemas/pulse.py`: add Pydantic models for each agent's structured output
- `backend/app/models/signal.py`: add `generated_code` field for audit trail
- `backend/app/config/prompts.yaml`: add new agent entries (pulse_coder, pulse_validator, pulse_viz, pulse_report_writer)
- `frontend/src/app/(workspace)/` routes: Overview page progress banner, toast notification, signal navigation, re-run flow
- `frontend/src/components/workspace/`: new progress banner component, updated signal cards, report button

</code_context>
