### Finding B — LLM Token Bloat Across Pipeline (major, fixed)
root_cause: |
  Three separate LLM calls received the full deep_profile JSON (1k-5k tokens per file) when only a fraction was needed:
  - Coder: received full profile JSON — needed only column names, dtypes, key stats
  - Viz: received full profile JSON — needed only column names + dtypes to avoid hallucinating column names
  - Viz retry: appended error context to already-bloated message instead of rebuilding clean
  With 7 signals and 2 files, deep_profile was serialized 15+ times into LLM prompts.
fix: |
  - _compact_profile_for_coder(): column names, dtypes, key numeric stats (min/max/mean), top categorical values, strong correlations (|r|≥0.5) — ~200-500 tokens vs 1-5k
  - _compact_profile_for_viz(): column names + dtypes only — ~100-200 tokens vs 1-5k
  - Viz retry: rebuild message fresh via _build_viz_message() helper instead of appending to bloated string
  - Report writer: also strip analysis_text (raw technical output) — finding field already contains interpreted business language
commit: "pending"
artifacts:

    issue: "deep_profile_str = json.dumps(full_profile, indent=2) passed to Coder and Viz for every signal"
