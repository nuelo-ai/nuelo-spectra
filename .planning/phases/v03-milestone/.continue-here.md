## This Session (6 bugs found and fixed)

### Bug 1: Stream Ownership — WelcomeScreen killed the stream on unmount
- **File:** `frontend/src/components/session/WelcomeScreen.tsx`
- **Root cause:** WelcomeScreen owned `useSSEStream` and called `startStream()`. When SessionPage detected messages and transitioned to ChatInterface, WelcomeScreen unmounted → useSSEStream cleanup (line 266-271) aborted the in-flight stream.
- **Fix:** Removed `useSSEStream` import/hook. Replaced with `useAddLocalMessage`. Auto-send useEffect and handleSend Case 3 now store `spectra_pending_stream` in sessionStorage + call `addLocalMessage()` (optimistic update triggers WelcomeScreen→ChatInterface transition).

### Bug 2: ChatInterface had no pending stream pickup
- **File:** `frontend/src/components/chat/ChatInterface.tsx`
- **Fix:** Added `pendingStreamHandled` ref + useEffect that reads `spectra_pending_stream` from sessionStorage on mount, parses it, removes it, calls `startStream()`.

### Bug 3: StreamEvent type missing top-level fields
- **File:** `frontend/src/types/chat.ts`
- **Root cause:** `StreamEvent` had `data?: Record<string, any>` but backend puts `generated_code`, `execution_result`, `analysis`, `follow_up_suggestions`, `search_sources`, `routing_decision` at TOP LEVEL of event object — not nested under `.data`.
- **Fix:** Removed `data` field. Added actual fields: `generated_code?`, `execution_result?`, `analysis?`, `final_response?`, `error?`, `follow_up_suggestions?`, `search_sources?`.

### Bug 4: getStreamingDataCard accessed event.data?.X — MAIN DATACARD BUG
- **File:** `frontend/src/components/chat/ChatInterface.tsx`
- **Root cause:** `getStreamingDataCard()` used `executionEvent?.data?.execution_result`, `codingEvent?.data?.generated_code`, etc. All always returned `undefined` because `.data` doesn't exist on events.
- **Fix:** Changed all to direct access: `executionEvent?.execution_result`, `codingEvent?.generated_code`, `analysisEvent?.analysis`, `analysisEvent?.follow_up_suggestions`, `analysisEvent?.search_sources`. Also fixed `isMemoryRoute` detection and memory route streaming section.

### Bug 5: hasStructuredNode had wrong node names
- **File:** `frontend/src/components/chat/ChatInterface.tsx`
- **Root cause:** Frontend checked `"coding"` but backend uses `"coding_agent"`. Checked `"data_analysis"` but backend uses `"da_response"`.
- **Fix:** `"coding"` → `"coding_agent"`, `"data_analysis"` → `"da_response"`. `"execute"` was already correct.

### Bug 6: ChatMessageResponse.file_id non-nullable — CRITICAL 500 BUG
- **File:** `backend/app/schemas/chat.py`
- **Root cause:** `file_id: UUID` was non-nullable in Pydantic schema. Session-based messages save with `file_id=None`. After stream completed, ChatInterface's refetch called `GET /sessions/{id}/messages` which returned **500** because Pydantic couldn't validate `None` → `UUID`. This is why the DataCard NEVER appeared post-streaming — the messages endpoint was broken for any session with stream responses.
- **Fix:** Changed to `file_id: UUID | None = None`.

<decisions_made>

- **Stream delegation pattern:** WelcomeScreen never owns streams. It stores pending stream info in sessionStorage (`spectra_pending_stream` key) and adds an optimistic message to TanStack Query cache. The optimistic message triggers SessionPage to transition to ChatInterface, which picks up the pending stream on mount.
- **sessionStorage keys:** `spectra_pending_stream` stores `{message, searchEnabled}` JSON (for stream handoff). Separate from existing `spectra_pending_message`/`spectra_pending_search` (for lazy session creation navigation bridge).
- **file_id nullable:** `file_id: UUID | None = None` with default for backward compat — old file-based messages have file_id set, new session-based messages have null.
- **Node name alignment:** Frontend node name constants must match backend graph node names exactly: `coding_agent`, `da_response`, `da_with_tools`, `execute`, `manager`, `code_checker`, `search_tools`, `halt`.

</decisions_made>
