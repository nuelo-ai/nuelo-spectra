## Phase Details

### Phase 55: Tier Restructure & Data Foundation
**Goal**: Users exist on the correct tier model with dual-balance credit tracking and all billing database tables ready for Stripe integration
**Depends on**: Phase 54 (previous milestone complete)
**Requirements**: TIER-01, TIER-02, TIER-03, TIER-04, TIER-05, TIER-06, TIER-07, TIER-08, DATA-01, DATA-02, DATA-03, DATA-04, DATA-05, DATA-06
**Success Criteria** (what must be TRUE):
  1. `user_classes.yaml` defines exactly 4 consumer tiers (Free Trial, On Demand, Basic, Premium) and the old `free`/`standard` keys are removed
  2. Existing users previously on `free` tier are migrated to the admin-configured target tier via Alembic data migration with zero broken lookups
  3. User credit balance is split into subscription credits and purchased credits, with subscription credits consumed first and purchased credits never reset on billing cycle
  4. APScheduler credit reset skips users on subscription tiers (Basic/Premium) and only resets On Demand / internal users
  5. Subscription, PaymentHistory, CreditPackage, and StripeEvent tables exist in the database with correct schemas via Alembic migration
**Plans**: 2 plans

Plans:
- [x] 55-01-PLAN.md — Config restructure, new billing models, Alembic migration
- [ ] 55-02-PLAN.md — Dual-balance CreditService refactor, registration trial setup, test updates

### Phase 56: Trial Expiration & Conversion Pressure
**Goal**: New users get a time-limited trial that blocks access after expiration, creating conversion pressure to select a paid plan
**Depends on**: Phase 55
**Requirements**: TRIAL-01, TRIAL-02, TRIAL-03, TRIAL-04, TRIAL-05, TRIAL-06, TRIAL-07
**Success Criteria** (what must be TRUE):
  1. New user registration sets Free Trial state with configurable credit amount and `trial_expires_at` set to registration date plus `trial_duration_days` platform setting
  2. Trial expiration is enforced in backend auth middleware covering web, API, and MCP paths -- expired trial users receive a 403 response regardless of access method
  3. Dashboard shows a trial banner with days remaining and credit balance that turns amber at 3 days or 10 credits remaining
  4. Expired trial users see a full blocking overlay with no access to any feature until they select a plan
  5. Trial users cannot purchase credit top-ups
**Plans**: 2 plans

Plans:
- [ ] 56-01-PLAN.md — Backend trial enforcement (auth dependency 402, UserResponse schema, top-up guard, tests) + frontend data layer (types, 402 handling, useTrialState hook)
- [ ] 56-02-PLAN.md — Trial UI components (TrialBanner, TrialExpiredOverlay) + layout integration + placeholder /settings/plan page

### Phase 57: Stripe Backend & Webhook Foundation
**Goal**: Backend payment infrastructure processes Stripe events reliably with idempotent webhooks and creates checkout sessions for subscriptions and one-time payments
**Depends on**: Phase 56
**Requirements**: STRIPE-01, STRIPE-02, STRIPE-03, STRIPE-04, STRIPE-05, STRIPE-06, STRIPE-07, STRIPE-08, STRIPE-09, STRIPE-10, SUB-06
**Success Criteria** (what must be TRUE):
  1. Stripe Python SDK is integrated with API keys configured via environment variables and all Stripe calls go through a `SubscriptionService`
  2. Stripe Checkout Sessions can be created for both subscription mode (Basic/Premium plans) and payment mode (credit top-up packages) with server-side price resolution (client never controls amount)
  3. Webhook endpoint at `/webhooks/stripe` verifies signatures, deduplicates events via `stripe_events` table, and processes `checkout.session.completed`, `invoice.paid`, `invoice.payment_failed`, `customer.subscription.updated`, and `customer.subscription.deleted`
  4. `invoice.paid` webhook resets subscription credits for the new billing cycle; `invoice.payment_failed` marks subscription at risk; `customer.subscription.deleted` downgrades user to On Demand
  5. Subscription state is stored locally in the Subscription table with Stripe as the source of truth
**Plans**: 3 plans

Plans:
- [x] 57-01-PLAN.md — Stripe SDK, config, exceptions, schemas, proxy fix, email templates, test scaffolds
- [x] 57-02-PLAN.md — SubscriptionService and webhook router with all event handlers
- [x] 57-03-PLAN.md — Checkout API endpoints, router registration, tests

### Phase 58: Billing UI & Subscription Management
**Goal**: Users can view plans, subscribe, manage their subscription, and purchase credit top-ups through an embedded Stripe checkout experience
**Depends on**: Phase 57
**Requirements**: SUB-01, SUB-02, SUB-03, SUB-04, SUB-05, TOPUP-01, TOPUP-02, TOPUP-03, TOPUP-04, TOPUP-05, BILL-01, BILL-02, BILL-03, BILL-04, BILL-05, BILL-06, BILL-07
**Success Criteria** (what must be TRUE):
  1. Plan Selection page (`/settings/plan`) displays On Demand, Basic, and Premium tiers with pricing and highlights the user's current plan or trial status
  2. User can subscribe to Basic or Premium via Stripe Embedded Checkout without leaving the Spectra domain, and the subscription activates on successful payment
  3. User can upgrade (immediate), downgrade (end of cycle), or cancel (end of cycle with purchased credits preserved) their subscription from the Manage Plan page
  4. User can purchase predefined credit packages (e.g., 50/200/500 credits) via Stripe Checkout, with purchased credits added to their balance on successful payment
  5. Manage Plan page (`/settings/billing`) shows subscription details, separate subscription/purchased credit balances, and billing history (payments, renewals, top-ups)
**Plans**: 4 plans

Plans:
- [x] 58-01-PLAN.md — Backend API endpoints (plans, change, cancel, on-demand, billing-history, packages)
- [x] 58-02-PLAN.md — Payment confirmation email templates and webhook email integration
- [x] 58-03-PLAN.md — Settings tab navigation restructure and Plan Selection page
- [x] 58-04-PLAN.md — Manage Plan page (billing) with all sections and post-redirect handling

### Phase 59: Admin Billing Tools
**Goal**: Admins can monitor, override, and manage user billing state including subscriptions, payments, refunds, and discount codes
**Depends on**: Phase 58
**Requirements**: ADMIN-01, ADMIN-02, ADMIN-03, ADMIN-04, ADMIN-05, ADMIN-06, ADMIN-07, ADMIN-08, ADMIN-09, ADMIN-10
**Success Criteria** (what must be TRUE):
  1. Admin can view a user's subscription status, payment history, and full Stripe billing event log from the user detail page
  2. Admin can force-set a user's tier regardless of Stripe state with mandatory reason logging, and can cancel a user's subscription on their behalf
  3. Admin can issue full or partial refunds for specific payments via the Stripe Refund API with corresponding credit adjustments
  4. Admin can configure billing platform settings: trial duration, credit prices, subscription prices, and subscription credit amounts
  5. Admin can create, manage, and deactivate discount codes that integrate with Stripe Coupons/Promotion Codes API
**Plans**: 4 plans

Plans:
- [x] 59-01-PLAN.md — Backend billing admin endpoints, StripeEvent migration, billing settings, refund + force-set tier service methods
- [x] 59-02-PLAN.md — Discount code service + router with Stripe Coupons/Promotion Codes sync, checkout promotion codes
- [x] 59-03-PLAN.md — Frontend billing tab, ForceSetTierDialog, RefundDialog, billing settings page, sidebar nav
- [x] 59-04-PLAN.md — Frontend discount codes page with create dialog and row actions
