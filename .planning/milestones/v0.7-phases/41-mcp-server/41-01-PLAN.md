<task type="auto">
  <name>Task 2: Create MCP server module with auth middleware and 6 tools</name>
  <files>backend/app/mcp_server.py</files>
  <action>
Create `backend/app/mcp_server.py` with the following components:

**1. Imports and Configuration:**
- Import FastMCP, Context from fastmcp
- Import Middleware, MiddlewareContext from fastmcp.server.middleware (verify exact import path at runtime — see research open question #2)
- Import get_http_headers from fastmcp.server.dependencies
- Import ToolError from fastmcp.exceptions
- Import ToolResult from fastmcp.tools.tool
- Import TextContent from mcp.types
- Import httpx, base64, json
- Import get_settings from app.config
- Create a `_get_base_url()` helper that reads `MCP_API_BASE_URL` from settings (defaults to `http://localhost:8000`)

**2. Auth Middleware — `ApiKeyAuthMiddleware(Middleware)`:**
- Override `on_call_tool(self, context, call_next)`:
  - Call `get_http_headers()` to get HTTP headers
  - Extract Authorization header, verify it starts with "Bearer "
  - Strip prefix to get the raw token
  - Verify token starts with "spe_" (Spectra API key prefix)
  - Store token in context via `context.fastmcp_context.set_state("api_key", token)` (verify exact API — may be `context.set_state()` depending on FastMCP version; check docs)
  - If auth fails at any step, raise `ToolError` with clear, actionable error messages (see research examples)
  - Call `await call_next(context)` to proceed

NOTE on `on_list_tools`: The CONTEXT.md says auth should be checked on tools/list too. If FastMCP middleware supports `on_list_tools`, implement it. If not, the tools themselves validate auth via httpx calls to the REST API, so unauthenticated tools/list only exposes tool names/descriptions — no data leakage. Document this in a code comment.

**3. FastMCP Instance:**
```python
mcp = FastMCP(
    name="Spectra Data Analysis",
    stateless_http=True,
    mask_error_details=True,
)
mcp.add_middleware(ApiKeyAuthMiddleware())
```

**4. Helper — `_api_headers(ctx: Context) -> dict`:**
Returns `{"Authorization": f"Bearer {ctx.get_state('api_key')}"}` for reuse across tools.

**5. Tool 1 — `spectra_upload_file(filename: str, file_content_base64: str, ctx: Context) -> str`:**
- Rich description per CONTEXT.md: explains what happens (AI onboarding, data brief, suggestions), supported formats (.csv, .xlsx, .xls), size limit (50MB)
- Annotations: `readOnlyHint=False, destructiveHint=False, openWorldHint=False`
- Decode base64 content (raise ToolError on decode failure)
- POST to `{base_url}/v1/files/upload` as multipart with `files={"file": (filename, file_bytes)}`
- Timeout: 120s (onboarding can take 10-45s per research)
- On success (201): return formatted markdown with File ID, filename, data brief, suggested questions
- On error: raise ToolError with error message from API response
- Addresses MCP-01

**6. Tool 2 — `spectra_run_analysis(file_id: str, question: str, ctx: Context) -> ToolResult`:**
- Rich description per CONTEXT.md: explains what it does, mentions credits cost, recommends spectra_get_context first
- Annotations: `readOnlyHint=True, openWorldHint=False`
- POST to `{base_url}/v1/chat/query` with JSON `{"query": question, "file_ids": [file_id]}`
- Timeout: 120s
- Return `ToolResult(content=[...])` with separate `TextContent` blocks for:
  - Analysis narrative (if present)
  - Generated Python code (if present, in ```python fenced block)
  - Chart specification (if present, in ```json fenced block)
  - Credits used note
- If no content blocks, add a fallback "Analysis completed but returned no content" block
- Addresses MCP-02

**7. Tool 3 — `spectra_list_files(ctx: Context) -> str`:**
- Concise description: "List all files uploaded to your Spectra account."
- GET to `{base_url}/v1/files`
- Timeout: 30s
- Return markdown table with File ID, Filename, Created date
- Handle empty list: "No files found. Upload a file first using spectra_upload_file."
- Addresses MCP-03

**8. Tool 4 — `spectra_delete_file(file_id: str, ctx: Context) -> str`:**
- Concise description: "Delete a file from your Spectra account by its file ID."
- DELETE to `{base_url}/v1/files/{file_id}`
- Timeout: 30s
- Return confirmation message
- Addresses MCP-03

**9. Tool 5 — `spectra_download_file(file_id: str, ctx: Context) -> str`:**
- Concise description: "Download a file's raw content by its file ID."
- GET to `{base_url}/v1/files/{file_id}/download`
- Timeout: 60s
- For text/CSV content: return file content as text (truncate at 10,000 chars with note per research pitfall #4)
- For binary content: return size and content type info
- Addresses MCP-03

**10. Tool 6 — `spectra_get_context(file_id: str, ctx: Context) -> str`:**
- Rich description per CONTEXT.md: explains what's returned (data brief, user context, suggestions), recommends using before spectra_run_analysis
- GET to `{base_url}/v1/files/{file_id}/context`
- Timeout: 30s
- Return formatted markdown with data brief, user context, and suggested questions
- Addresses MCP-04

**11. Factory Function — `create_mcp_app()`:**
```python
def create_mcp_app():
    """Create and return the MCP ASGI application for mounting."""
    return mcp.http_app(path="/")
```

**Error Handling Pattern (all tools):**
- Wrap httpx calls in try/except
- On httpx errors (connection, timeout): raise ToolError with clear message
- On non-success HTTP status: extract error from API response JSON envelope (`response.json().get("error", {}).get("message")`) and raise ToolError
- The `mask_error_details=True` on the FastMCP instance acts as a safety net for unhandled exceptions

**Important implementation notes:**
- All tools MUST be `async def` (not sync — would fail in ASGI context per research anti-patterns)
- Do NOT import service functions directly — always call REST API via httpx to preserve auth/credit/logging chain
- Use `httpx.AsyncClient` (not `httpx.Client`)
- The `ctx: Context` parameter is how FastMCP injects the tool context (where we stored the api_key)
