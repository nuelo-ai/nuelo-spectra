Create `backend/app/routers/api_v1/files.py` with `router = APIRouter(prefix="/files", tags=["API v1 - Files"])`.

Use `ApiAuthUser` and `DbSession` from `app.dependencies`. Use `ApiResponse`, `api_error` from `app.routers.api_v1.schemas`.

**Endpoint 1: `POST /files/upload` (APIF-01)**
- Accept `file: UploadFile` (multipart form upload)
- Validate extension: `.csv`, `.xlsx`, `.xls` only. Return `api_error(400, "INVALID_FILE_TYPE")` on mismatch.
- Validate file size: check `file.size` against `MAX_FILE_SIZE_MB` from settings (existing constant in `app.config`). Return `api_error(400, "FILE_TOO_LARGE")` if exceeded.
- Call `FileService.upload_file(db, user.id, file, ext)` — reuse existing method from `backend/app/services/file.py`. Check the existing frontend `files.py` router to see exactly how it calls this method and replicate the same pattern.
- **Synchronous onboarding** (per user decision): `await agent_service.run_onboarding(db, file_record.id, user.id, "")` — do NOT use `asyncio.create_task`. This blocks until onboarding completes.
- After onboarding, `await db.refresh(file_record)` to get populated `data_summary` and `query_suggestions`.
- Return status 201 with `ApiResponse(data={"id": str(file_record.id), "filename": file_record.original_filename, "data_brief": file_record.data_summary, "query_suggestions": file_record.query_suggestions, "created_at": file_record.created_at.isoformat()})`.
- Wrap onboarding in try/except: if onboarding fails, the file is still saved (it can be queried later when suggestions are available). Return `api_error(500, "ONBOARDING_FAILED")` but include the file_id in the error message so the caller knows the file was uploaded.

**Endpoint 2: `GET /files` (APIF-02)**
- Call `FileService.list_user_files(db, user.id)` — check existing service for the exact method name.
- Return `ApiResponse(data=[{"id": str(f.id), "filename": f.original_filename, "created_at": f.created_at.isoformat(), "updated_at": f.updated_at.isoformat() if f.updated_at else None} for f in files])`.
- No pagination per user decision.

**Endpoint 3: `DELETE /files/{file_id}` (APIF-03)**
- `file_id: UUID` path parameter.
- Call `FileService.get_user_file(db, user.id, file_id)` to verify ownership. If None, return `api_error(404, "FILE_NOT_FOUND")`.
- Call `FileService.delete_file(db, file_id)` — check existing service for exact method signature.
- Return `ApiResponse(data={"deleted": True})` with status 200.

**Endpoint 4: `GET /files/{file_id}/download` (APIF-04)**
- `file_id: UUID` path parameter.
- Call `FileService.get_user_file(db, user.id, file_id)` for ownership check. If None, return `api_error(404, "FILE_NOT_FOUND")`.
- Return `FileResponse` (from `fastapi.responses`) pointing to the file's stored path. Set `filename=file_record.original_filename` and appropriate `media_type`. This is the ONE endpoint that does NOT wrap in ApiResponse (binary stream, per research pitfall #7).

Check `backend/app/routers/files.py` (existing frontend router) for the exact FileService method names, import paths, and file path resolution pattern. Mirror those patterns.

**2a. Create `backend/app/routers/api_v1/context.py`:**

`router = APIRouter(prefix="/files", tags=["API v1 - File Context"])` — note: context routes are nested under `/files/{file_id}/...` path, so they share the `/files` prefix with files.py.

Actually, to avoid router prefix conflicts, use `prefix=""` and include the full path in each endpoint decorator. OR use a different approach: put context endpoints in files.py itself.

**Better approach:** Since both files.py and context.py use `/files` prefix, and FastAPI allows multiple routers on the same prefix, keep them separate for code organization but both use `prefix="/files"`.

Endpoints:

**Endpoint 1: `GET /files/{file_id}/context` (APIC-01)**
- `file_id: UUID` path parameter.
- Call `FileService.get_user_file(db, user.id, file_id)`. If None, return `api_error(404, "FILE_NOT_FOUND")`.
- Return `ApiResponse(data={"id": str(f.id), "filename": f.original_filename, "data_brief": f.data_summary, "data_summary": f.data_summary, "user_context": f.user_context, "query_suggestions": f.query_suggestions, "created_at": f.created_at.isoformat()})`.
- Note: `data_brief` and `data_summary` are the same field (data_summary stores the AI-generated brief). Include both keys for clarity to API consumers.

**Endpoint 2: `PUT /files/{file_id}/context` (APIC-02)**
- `file_id: UUID` path parameter.
- Request body Pydantic model `UpdateContextRequest`: `data_summary: str | None = None`, `user_context: str | None = None`. Both optional — caller can update one or both.
- Call `FileService.get_user_file(db, user.id, file_id)`. If None, return `api_error(404, "FILE_NOT_FOUND")`.
- Update the fields that are provided (not None). Use direct attribute assignment on the file record then `await db.flush()`.
- Return `ApiResponse(data={"id": str(f.id), "data_summary": f.data_summary, "user_context": f.user_context})`.

**Endpoint 3: `GET /files/{file_id}/suggestions` (APIC-03)**
- `file_id: UUID` path parameter.
- Call `FileService.get_user_file(db, user.id, file_id)`. If None, return `api_error(404, "FILE_NOT_FOUND")`.
- Return `ApiResponse(data={"id": str(f.id), "query_suggestions": f.query_suggestions or []})`.
