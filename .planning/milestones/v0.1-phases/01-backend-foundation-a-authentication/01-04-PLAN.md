Two changes are needed:

1. In `backend/.env`, set EMAIL_SERVICE_API_KEY to empty string:
   Change: `EMAIL_SERVICE_API_KEY=dev-api-key`
   To: `EMAIL_SERVICE_API_KEY=`

2. In `backend/app/services/email.py`, make dev mode detection explicit so it cannot break again if someone sets a placeholder value. Replace the falsy check on line 32:
   ```python
   if not settings.email_service_api_key:
   ```
   With an explicit dev mode check that also catches common placeholder values:
   ```python
   _DEV_PLACEHOLDERS = {"", "dev-api-key", "your-email-api-key", "changeme"}
   ```
   Define `_DEV_PLACEHOLDERS` as a module-level set above the function (after the logger line). Then change the condition to:
   ```python
   if not settings.email_service_api_key or settings.email_service_api_key.strip() in _DEV_PLACEHOLDERS:
   ```

   This ensures dev mode activates when:
   - API key is empty, None, or missing (original behavior)
   - API key matches known placeholder strings (fixes the bug)

   Do NOT change anything else in the file. The Mailgun production path, HTML template, error handling, and function signature all remain unchanged.

Start the backend server and run the complete forgot-password and reset-password flow to verify both UAT Test 7 and Test 8 pass.

Steps:
1. Ensure PostgreSQL is running (Docker container)
2. Start the FastAPI backend: `cd /Users/marwazisiagian/Documents/ms-dev/spectra-project/spectra-dev/backend && uv run uvicorn app.main:app --reload --log-level info`
3. Wait for startup, then in a separate terminal:

   a. Create a test user (or use existing):
      ```bash
      curl -s -X POST http://localhost:8000/auth/signup \
        -H "Content-Type: application/json" \
        -d '{"email":"resettest@example.com","password":"testpass123"}' | python3 -m json.tool
      ```
      (If 409 returned, user already exists -- that is fine)

   b. Call forgot-password:
      ```bash
      curl -s -X POST http://localhost:8000/auth/forgot-password \
        -H "Content-Type: application/json" \
        -d '{"email":"resettest@example.com"}' -w "\nHTTP Status: %{http_code}\n"
      ```
      Expected: HTTP 202. Check the backend server logs for the PASSWORD RESET REQUEST block with the reset link.

   c. Extract the reset token from the console log URL (the token parameter in the reset link).

   d. Call reset-password with the token:
      ```bash
      curl -s -X POST http://localhost:8000/auth/reset-password \
        -H "Content-Type: application/json" \
        -d '{"token":"TOKEN_FROM_LOGS","new_password":"newpass456"}' -w "\nHTTP Status: %{http_code}\n"
      ```
      Expected: HTTP 200.

   e. Verify login with new password:
      ```bash
      curl -s -X POST http://localhost:8000/auth/login \
        -H "Content-Type: application/json" \
        -d '{"email":"resettest@example.com","password":"newpass456"}' -w "\nHTTP Status: %{http_code}\n"
      ```
      Expected: HTTP 200 with access_token and refresh_token.

   f. Verify old password fails:
      ```bash
      curl -s -X POST http://localhost:8000/auth/login \
        -H "Content-Type: application/json" \
        -d '{"email":"resettest@example.com","password":"testpass123"}' -w "\nHTTP Status: %{http_code}\n"
      ```
      Expected: HTTP 401.

4. Stop the backend server after verification.

If the server is already running, skip step 2 and just run the curl commands.
