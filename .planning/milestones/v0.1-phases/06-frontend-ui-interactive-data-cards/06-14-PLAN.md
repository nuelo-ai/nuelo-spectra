Two surgical fixes, both diagnosed with exact line numbers:

**Fix 1: Keep useFileSummary enabled in ready state (FileUploadZone.tsx line 32)**

The useFileSummary hook currently passes null when uploadStage is not "analyzing":
```
const { data: summary } = useFileSummary(uploadStage === "analyzing" ? uploadedFileId : null);
```

When the useEffect (lines 38-44) transitions uploadStage from "analyzing" to "ready", the hook receives null on the next render. This disables the React Query query (enabled: !!fileId), causing React Query to clear the cached summary data. The conditional render at line 200 checks `uploadStage === "ready" && summary?.data_summary` but summary is now undefined.

Change line 32 to:
```typescript
const { data: summary } = useFileSummary(
  uploadStage === "analyzing" || uploadStage === "ready" ? uploadedFileId : null
);
```

This keeps the query enabled through the ready state so summary data persists for the analysis display and Continue to Chat button.

**Fix 2: Replace invalidateQueries with refetchQueries in useUploadFile (useFileManager.ts line 55)**

The useUploadFile mutation's onSuccess only calls invalidateQueries(), which marks the ["files"] query as stale but does NOT force an immediate refetch for already-mounted components like FileSidebar. TanStack Query invalidation only triggers refetch on next mount or window focus, not immediately.

In useFileManager.ts, change the useUploadFile onSuccess (line 53-56) from:
```typescript
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: ["files"] });
},
```
To:
```typescript
onSuccess: () => {
  queryClient.refetchQueries({ queryKey: ["files"] });
},
```

refetchQueries() forces an immediate refetch for all active observers of the query, ensuring FileSidebar updates right away.

**Fix 3: Remove redundant manual invalidation from FileUploadZone.tsx line 79**

Inside the onDrop callback's onSuccess handler (lines 72-94), line 79 manually calls:
```typescript
queryClient.invalidateQueries({ queryKey: ["files"] });
```

This is redundant because the mutation's own onSuccess (Fix 2 above) already handles the refetch. Remove this line entirely to avoid confusing double-invalidation. The mutation handles it.

**Do NOT change anything else.** The useEffect for state transition (lines 38-44), the Continue to Chat button handler (lines 207-228), and all other logic remain exactly as-is. These three line-level changes are sufficient.
