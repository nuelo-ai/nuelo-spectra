# Phase 52.1: Delete and Rename Collection - Research

**Researched:** 2026-03-09
**Domain:** Next.js 14 workspace UI + FastAPI collection endpoints
**Confidence:** HIGH

## Summary

This phase is a well-scoped UI + API feature that extends the existing Collection CRUD surface. All frontend primitives are already present in the codebase (shadcn DropdownMenu, Dialog, TanStack Query mutations, sonner toasts). The PATCH endpoint and `update_collection` service method are fully implemented. The DELETE endpoint is missing — no router endpoint and no `delete_collection` service method exist yet; both must be created.

The rename flow reuses `CreateCollectionDialog` with an edit mode toggle, eliminating a new component. The delete flow is a fresh `AlertDialog`-style confirmation dialog (can use the standard shadcn `Dialog` with destructive button). State for both dialogs is managed in the list page (`workspace/page.tsx`) for card-triggered actions and locally in the detail page (`collections/[id]/page.tsx`) for header-triggered actions.

**Primary recommendation:** Build backend first (add `delete_collection` service + DELETE endpoint), then frontend (edit mode in existing dialog + new delete confirmation dialog + kebab menus in both surfaces). No DB migration required.

### Pattern 1: Backend DELETE endpoint (follows remove-file pattern)

The `DELETE /{collection_id}/files/{file_id}` endpoint in `collections.py` (lines 308-324) is the direct model. The new collection delete follows the same shape: ownership check via service, `await db.commit()`, return 204.

```python
# Pattern from existing remove_file_from_collection endpoint:
@router.delete("/{collection_id}", status_code=status.HTTP_204_NO_CONTENT)
async def delete_collection(
    collection_id: UUID,
    current_user: WorkspaceUser,
    db: DbSession,
) -> None:
    """Delete a collection and all cascade children (signals, reports, pulse_runs)."""
    deleted = await CollectionService.delete_collection(
        db, collection_id, current_user.id
    )
    await db.commit()
    if not deleted:
        raise HTTPException(
            status_code=status.HTTP_404_NOT_FOUND,
            detail="Collection not found",
        )
```

### Pattern 2: Backend delete_collection service method (follows remove_file_from_collection)

```python
# Source: existing CollectionService.remove_file_from_collection() pattern
@staticmethod
async def delete_collection(
    db: AsyncSession,
    collection_id: UUID,
    user_id: UUID,
) -> bool:
    """Delete a collection (cascade handled by DB FK ondelete=CASCADE).

    Returns True if deleted, False if not found / not owned.
    """
    stmt = select(Collection).where(
        Collection.id == collection_id,
        Collection.user_id == user_id,
    )
    result = await db.execute(stmt)
    collection = result.scalar_one_or_none()

    if collection is None:
        return False

    await db.delete(collection)
    await db.flush()
    return True
```

**No DB migration needed** — `ondelete="CASCADE"` already exists on all child FKs (signals, collection_files, pulse_runs, reports) per CONTEXT.md specifics.

### Pattern 3: Frontend useDeleteCollection hook (follows useRemoveFile)

```typescript
// Source: existing useRemoveFile pattern in useWorkspace.ts (lines 196-214)
export function useDeleteCollection() {
  const queryClient = useQueryClient();
  return useMutation<void, Error, string>({
    mutationFn: async (collectionId) => {
      const response = await apiClient.delete(`/collections/${collectionId}`);
      if (!response.ok) throw new Error(`Failed: ${response.status}`);
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["collections"] });
    },
  });
}
```

### Pattern 4: Frontend useUpdateCollection hook (follows useCreateCollection)

```typescript
// Source: existing useCreateCollection pattern in useWorkspace.ts (lines 124-139)
export function useUpdateCollection(collectionId: string) {
  const queryClient = useQueryClient();
  return useMutation<CollectionDetail, Error, { name: string; description?: string }>({
    mutationFn: async (body) => {
      const response = await apiClient.patch(`/collections/${collectionId}`, body);
      if (!response.ok) {
        const errorData = await response.json().catch(() => ({}));
        throw new Error(errorData.detail || `Failed: ${response.status}`);
      }
      return response.json();
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["collections"] });
      queryClient.invalidateQueries({ queryKey: ["collections", collectionId] });
    },
  });
}
```

### Pattern 5: Edit mode in CreateCollectionDialog

The existing dialog has no `collection` prop. The edit mode toggle adds:
- `collection?: CollectionListItem` prop to `CreateCollectionDialogProps`
- When `collection` is present: pre-fill `name`/`description` state from prop, change `DialogTitle` to "Edit Collection", change submit button label to "Save", call `useUpdateCollection` instead of `useCreateCollection`
- Reset state on dialog close (same as existing `handleCancel` pattern)
- On success: close dialog and invalidate queries (no router.push — stay on same page)

### Pattern 6: Kebab menu on CollectionCard

The card is currently wrapped in a `<Link>`. Adding a `DropdownMenu` inside a `<Link>` requires `e.preventDefault()` / `e.stopPropagation()` on the trigger to prevent card navigation. The cleanest approach is to restructure: remove the outer `<Link>` wrapper and instead put the `<Link>` inside the `CardContent`, making only the card body (not the whole card) navigable — or alternatively keep the outer Link and stop propagation on the dropdown trigger button.

Looking at existing patterns in `ChatListItem.tsx` and `MyFilesTable.tsx` which use `DropdownMenu` inside clickable rows, stop-propagation on the trigger is the established pattern.

```typescript
// DropdownMenuTrigger inside Link card — stop propagation pattern:
<Button
  variant="ghost"
  size="icon"
  className="h-6 w-6"
  onClick={(e) => e.preventDefault()}  // prevent Link navigation
>
  <MoreVertical className="h-4 w-4" />
</Button>
```

The CollectionCard will need `onRename` and `onDelete` callback props passed from the parent.

### Pattern 7: State management in workspace/page.tsx

```typescript
// Add to WorkspacePage:
const [renameTarget, setRenameTarget] = useState<CollectionListItem | null>(null);
const [deleteTarget, setDeleteTarget] = useState<CollectionListItem | null>(null);

// Pass down to CollectionList → CollectionCard:

// Render dialogs at page level:

  onOpenChange={(open) => { if (!open) setRenameTarget(null); }}
  collection={renameTarget ?? undefined}

  open={deleteTarget !== null}
  onOpenChange={(open) => { if (!open) setDeleteTarget(null); }}
  collection={deleteTarget}
/>
```

### Pattern 8: Delete confirmation dialog (new component)

A standalone `DeleteCollectionDialog` component (separate from `CreateCollectionDialog`) is the clean choice — it has completely different content and no shared logic.

```typescript
interface DeleteCollectionDialogProps {
  open: boolean;
  onOpenChange: (open: boolean) => void;
  collection: CollectionListItem | null;
  onSuccess?: () => void;  // optional: for detail page post-delete navigation
}
```

Dialog body: collection name in `<strong>` + cascade warning text. Footer: "Cancel" ghost button + "Delete" destructive button disabled when `isPending`, labeled "Deleting..." during pending.

### Anti-Patterns to Avoid

- **Wrapping the entire Card in a Link when adding a DropdownMenu inside:** The DropdownMenu click will bubble up and trigger navigation. Use `e.preventDefault()` on the trigger, or restructure the card so only the non-interactive area is wrapped in Link.
- **Creating a separate EditCollectionDialog:** Locked decision — reuse `CreateCollectionDialog` with an optional `collection` prop.
- **Optimistic update for delete/rename:** Locked decision — wait for server confirmation then invalidate queries.
- **Calling `router.push('/workspace')` from the list page on delete:** Only the detail page does this; the list page uses query invalidation only.
- **Forgetting to invalidate both `['collections']` and `['collections', id]` on rename:** The detail page header re-renders the collection name from the detail query key.

## Common Pitfalls

### Pitfall 1: Link + DropdownMenu click conflict
**What goes wrong:** Clicking the DropdownMenuTrigger button navigates to the collection detail page AND opens the dropdown simultaneously.
**Why it happens:** The `<Link>` wraps the entire `<Card>`, so all click events inside it bubble to the Link.
**How to avoid:** Add `onClick={(e) => e.preventDefault()}` to the `DropdownMenuTrigger` `asChild` button. Or restructure the card to have the Link wrap only the non-interactive zone.
**Warning signs:** Clicking ⋮ causes page navigation.

### Pitfall 2: delete_collection service method missing
**What goes wrong:** CONTEXT.md says "CollectionService.delete_collection() exists at line ~255" but a code search confirms this method does NOT exist in the current `collection.py`. The router endpoint is also missing.
**Why it happens:** This was listed as a reusable asset in the context doc, but it hasn't been implemented yet.
**How to avoid:** Wave 0 / Plan 1 must create both the service method and the router endpoint before any frontend work calls DELETE.
**Warning signs:** `AttributeError: type object 'CollectionService' has no attribute 'delete_collection'`.

### Pitfall 3: Dialog state not reset on close
**What goes wrong:** Opening rename dialog for collection "A", closing it, then opening for collection "B" — the pre-filled name still shows "A".
**Why it happens:** `useState` inside a Dialog doesn't reset when `open` changes to false unless explicitly cleared.
**How to avoid:** Use a `useEffect` inside `CreateCollectionDialog` that resets `name`/`description` state whenever the `collection` prop changes, or whenever `open` changes to `true` with a new collection. Or mount/unmount the dialog content with a `key={collection?.id}` prop.
**Warning signs:** Stale values appear in form fields.

### Pitfall 4: Missing detail query invalidation after rename
**What goes wrong:** Renaming a collection from the detail page updates the list but the `<h1>` heading in the detail page header still shows the old name.
**Why it happens:** The detail page reads from `useCollectionDetail(id)` which uses query key `['collections', id]` — if only `['collections']` is invalidated, the detail query is not refreshed.
**How to avoid:** `useUpdateCollection` must invalidate both `['collections']` (list) and `['collections', collectionId]` (detail). Confirmed in Pattern 4 above.
**Warning signs:** Collection name in header doesn't update after rename.

### Pitfall 5: Cascade assumption without DB verification
**What goes wrong:** Assuming DB cascade handles orphan cleanup but child rows remain.
**Why it happens:** If any FK was defined without `ondelete="CASCADE"`.
**How to avoid:** The CONTEXT.md specifics confirm cascade is already in place for signals, pulse_runs, and reports. No action needed — but verify during QA by checking child rows are gone post-delete.
**Warning signs:** Orphaned signals/reports remain after collection deletion.
