  <action>
    Generate migration using `cd backend && alembic revision --autogenerate -m "add admin portal tables and user fields"`.

    After generation, review and modify the migration to follow the three-step pattern (per DB-08):

    **Step 1 -- Add columns to users table:**
    ```python
    op.add_column("users", sa.Column("is_admin", sa.Boolean(), server_default=sa.text("false"), nullable=False))
    op.add_column("users", sa.Column("user_class", sa.String(20), server_default="free", nullable=False))
    ```
    CRITICAL: Use `server_default` so existing rows get backfilled automatically by PostgreSQL. Without this, the migration fails if the users table has existing rows.

    **Step 2 -- Create 5 new tables:**
    Let Alembic autogenerate handle the `op.create_table()` calls for: platform_settings, user_credits, credit_transactions, invitations, admin_audit_log. Verify all indexes and constraints are present.

    **Step 3 -- Backfill credit records:**
    After the tables are created, add explicit backfill SQL:
    ```python
    op.execute("""
        INSERT INTO user_credits (id, user_id, balance, created_at)
        SELECT gen_random_uuid(), id, 0, NOW()
        FROM users
        WHERE id NOT IN (SELECT user_id FROM user_credits)
    """)
    ```
    This creates a user_credits row with balance=0 for every existing user that doesn't already have one (idempotent).

    **Downgrade**: The downgrade function should:
    1. Drop tables in reverse dependency order (admin_audit_log, invitations, credit_transactions, user_credits, platform_settings)
    2. Drop columns: `op.drop_column("users", "user_class")` then `op.drop_column("users", "is_admin")`

    Review the generated migration carefully. Ensure:
    - All UUID primary keys use `sa.UUID()` type (not `sa.Uuid()` or string)
    - JSONB column uses `postgresql.JSONB`
    - NUMERIC columns use `sa.NUMERIC(10, 1)`
    - ForeignKey constraints reference "users.id" correctly
    - All expected indexes are created
