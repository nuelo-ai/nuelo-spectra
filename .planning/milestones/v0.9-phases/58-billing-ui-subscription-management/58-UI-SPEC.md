## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (text-sm) | 400 (normal) | 1.5 (leading-normal) |
| Label | 14px (text-sm) | 400 (normal) | 1.4 |
| Heading | 20px (text-xl) | 600 (semibold) | 1.2 |
| Display | 28px (text-2xl / text-3xl) | 600 (semibold) | 1.2 |

Notes:
- Page title ("Choose Your Plan", "Manage Plan") uses Display at 28px semibold — consistent with existing settings page pattern but scoped to the content area (no gradient text on sub-pages)
- Tier card names use Heading at 20px semibold
- Pricing text on tier cards uses 28px semibold (Display) for visual emphasis
- Feature lists, billing history rows, credit balances use Body at 14px normal
- Section labels ("Credits", "Billing History") use Label at 14px normal with `uppercase tracking-wide` for visual distinction without an additional weight
- Only two font weights used: 400 (normal) for body and label text, 600 (semibold) for headings and display text

## Interaction Contracts

### Subscribe Flow
1. User clicks "Subscribe to Basic" or "Subscribe to Premium" on tier card
2. Button shows loading spinner (disabled state)
3. Frontend calls `POST /api/subscriptions/checkout` with `{ tier: "basic" | "premium" }`
4. Backend returns `{ checkout_url: "https://checkout.stripe.com/..." }`
5. Frontend redirects via `window.location.href = checkout_url`
6. After payment, Stripe redirects to `/settings/billing?session_id={id}`
7. Page detects `session_id` query param, shows success toast: "Subscription activated!"
8. React Query invalidates user and subscription data

### Select On Demand Flow
1. User clicks "Select On Demand"
2. Button shows loading spinner
3. Frontend calls `POST /api/subscriptions/select-on-demand`
4. On success, redirect to `/settings/billing` with success toast: "Plan updated to On Demand"

### Upgrade/Downgrade Flow
1. User clicks "Change Plan" on billing page
2. Navigate to `/settings/plan`
3. User selects a different plan
4. AlertDialog opens with confirmation copy (see Copywriting Contract)
5. User confirms -> loading state on confirm button
6. Frontend calls `POST /api/subscriptions/change` with `{ tier: "new_tier" }`
7. On success, redirect to `/settings/billing` with toast
8. Upgrade toast: "Plan upgraded to Premium! New credits added."
9. Downgrade toast: "Plan will change to Basic at end of billing cycle."

### Cancel Flow
1. User clicks "Cancel Plan" (outline destructive button)
2. AlertDialog opens (see Copywriting Contract for content)
3. User clicks "Cancel Plan" in dialog -> loading state
4. Frontend calls `POST /api/subscriptions/cancel`
5. On success, plan status card updates to show "Cancelled — active until [date]"
6. Toast: "Subscription cancelled. Access continues until [date]."

### Credit Top-Up Flow
1. User clicks "Buy Credits" on a credit package card
2. Button shows loading spinner
3. Frontend calls `POST /api/credits/purchase` with `{ package_id: "..." }`
4. Backend returns `{ checkout_url: "https://checkout.stripe.com/..." }`
5. Frontend redirects to Stripe Checkout
6. After payment, Stripe redirects to `/settings/billing?topup=success`
7. Page detects `topup=success` query param, shows success toast: "Credits added to your balance!"

## Settings Navigation Contract

Tab navigation using shadcn `Tabs` component:

| Tab Label | Route | Contains |
|-----------|-------|----------|
| Profile | /settings | ProfileForm, PasswordForm, AccountInfo |
| API Keys | /settings/keys | ApiKeySection |
| Plan & Billing | /settings/plan or /settings/billing | Plan Selection or Manage Plan |

Implementation approach:
- Settings layout wraps all `/settings/*` pages
- Tabs component at top, each tab is a Next.js Link
- Active tab determined by `usePathname()`
- "Plan & Billing" tab highlights for both `/settings/plan` and `/settings/billing`
- Tab bar is horizontally scrollable on mobile (if needed)
