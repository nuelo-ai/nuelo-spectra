<context>
The user asked for an architectural assessment of decoupling chat from files.

Key findings from the deep dive:

**Coupling severity: 9/10** — "one chat = one file" is baked into every layer.

**Backend coupling (6 points):**
- DB: ChatMessage.file_id is NOT NULL FK with CASCADE
- Routes: All 6 endpoints use /{file_id}/... URL pattern
- Thread IDs: f"file_{file_id}_user_{user_id}" encodes file isolation
- Agent state: file_id, file_path, data_summary — all singular fields
- Agent service: Loads one file_record, passes one file's data to graph
- Chat service: All queries filter by file_id

**Frontend coupling (~15 components, ~8 hooks):**
- Tab store: FileTab { fileId, fileName } — tab = file
- Sidebar: Clicking file opens chat tab (no conversation concept)
- ChatInterface: All props, hooks, API calls parameterized by fileId
- Upload flow: Auto-opens chat tab after upload
- Cache keys: All include fileId as primary dimension
- Types: ChatMessageResponse.file_id on every message

**The change requires:**
1. New Conversation entity (DB model + junction table for many-to-many)
2. New /conversations/{conv_id}/... API routes
3. Agent state refactor: file_paths: list, data_summaries: dict
4. Frontend paradigm shift: tabs = conversations, not files
5. New UI: file picker/attacher, conversation list, attached files bar
6. Multi-file code generation (hardest part — multiple DataFrames in E2B)

**Estimated scope: 4-5 phases**, with frontend being ~half the work.
The hardest pieces: multi-file agent intelligence and frontend UX paradigm shift.
</context>
